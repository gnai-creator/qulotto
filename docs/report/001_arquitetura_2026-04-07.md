# Proposta de Arquitetura do Qulotto

## Objetivo

Construir um projeto de software orientado a estudo que explore se técnicas inspiradas em computação quântica e computação quântica podem ser usadas para gerar e ranquear combinações de números da Lotofácil, mantendo o sistema cientificamente honesto, modular e fácil de evoluir.

O projeto não deve assumir que computação quântica consegue "prever" resultados de loteria. Em vez disso, ele deve tratar o problema como:

- geração de combinações;
- ranqueamento heurístico;
- simulação e backtesting;
- comparação entre abordagens clássicas e quânticas.

## Princípios Arquiteturais

- Separar lógica de domínio da lógica experimental.
- Tratar módulos quânticos como motores de estratégia plugáveis.
- Manter ingestão de dados, pontuação e avaliação determinísticas e testáveis.
- Tornar todo experimento reproduzível.
- Preferir pipelines explicáveis em vez de alegações opacas de predição ponta a ponta.

## Módulos de Alto Nível

### 1. Núcleo de Domínio

Responsabilidade:

- representar as regras da Lotofácil;
- validar jogos;
- calcular acertos em relação aos resultados oficiais;
- expor primitivas reutilizáveis para o restante do sistema.

Escopo sugerido:

- `src/core/lotofacil.py`
- `src/core/game.py`
- `src/core/validation.py`
- `src/core/scoring.py`

Entidades principais:

- `Draw`: resultado oficial de um concurso;
- `Ticket`: jogo gerado com 15 números;
- `Pool`: conjunto de jogos candidatos;
- `Score`: metadados de ranqueamento de um jogo.

### 2. Camada de Dados

Responsabilidade:

- importar sorteios históricos oficiais;
- normalizar e versionar datasets;
- expor acesso adequado aos dados para os experimentos.

Escopo sugerido:

- `src/data/loaders.py`
- `src/data/models.py`
- `src/data/repository.py`
- `data/raw/`
- `data/processed/`

Observações:

- Manter dados brutos imutáveis.
- Datasets processados devem ser reproduzíveis a partir dos dados brutos.
- Cada geração de dataset deve registrar fonte, data e etapas de transformação.

### 3. Camada de Engenharia de Features

Responsabilidade:

- derivar features estatísticas clássicas a partir dos sorteios passados;
- transformar dados históricos em entradas para estratégias de ranqueamento.

Exemplos de features:

- frequência de cada número;
- atraso de recorrência;
- equilíbrio entre pares e ímpares;
- padrões de números consecutivos;
- distribuição por faixas numéricas;
- sobreposição com concursos recentes.

Escopo sugerido:

- `src/features/frequency.py`
- `src/features/patterns.py`
- `src/features/encoders.py`

Essa camada é importante porque tanto estratégias clássicas quanto quânticas podem consumir o mesmo conjunto normalizado de features.

### 4. Camada de Estratégias

Responsabilidade:

- gerar ou ranquear jogos candidatos;
- encapsular abordagens alternativas atrás de uma interface comum.

Interface sugerida:

- `generate_candidates(context) -> list[Ticket]`
- `rank_candidates(candidates, context) -> list[Score]`

Submódulos:

- `src/strategies/random_baseline.py`
- `src/strategies/statistical_baseline.py`
- `src/strategies/quantum_inspired.py`
- `src/strategies/quantum_hybrid.py`

Essa é a abstração central do projeto. O restante da arquitetura não deve se importar se uma estratégia é aleatória, clássica ou quântica.

### 5. Camada Quântica

Responsabilidade:

- implementar algoritmos quânticos ou inspirados em computação quântica de forma isolada;
- traduzir a saída dos experimentos em sinais de ranqueamento dos jogos.

Escopo sugerido:

- `src/quantum/circuits.py`
- `src/quantum/encodings.py`
- `src/quantum/samplers.py`
- `src/quantum/adapters.py`

Possíveis trilhas de estudo:

- amplitude encoding de features selecionadas;
- circuitos variacionais para pontuação de padrões;
- amostragem quântica para exploração de candidatos;
- busca inspirada em QAOA sobre espaços de candidatos com restrições;
- loops híbridos de ranqueamento clássico-quântico.

Restrição importante:

A camada quântica não deve conter regras de negócio da Lotofácil. Ela deve apenas produzir probabilidades, pontuações ou amostras que a camada de estratégias consiga interpretar.

### 6. Camada de Avaliação e Pesquisa

Responsabilidade:

- executar backtests das estratégias contra concursos históricos;
- comparar desempenho entre diferentes métodos;
- gerar relatórios de pesquisa.

Escopo sugerido:

- `src/evaluation/backtest.py`
- `src/evaluation/metrics.py`
- `src/evaluation/reporting.py`

Métricas:

- média de acertos;
- distribuição de acertos;
- qualidade dos melhores candidatos em top-k;
- estabilidade ao longo de janelas temporais;
- comparação contra baseline aleatória.

Essa camada é obrigatória se o projeto for educacional em vez de apenas especulativo.

### 7. Camada de Aplicação

Responsabilidade:

- fornecer uma interface inicial em CLI para executar experimentos e gerar relatórios.

Escopo sugerido:

- `src/app/cli.py`
- `src/app/config.py`

Comandos iniciais:

- carregar dataset;
- gerar jogos candidatos;
- rodar backtest;
- comparar estratégias;
- exportar relatório.

Uma CLI é melhor como primeira interface do que uma aplicação web, porque mantém o projeto focado em aprendizado e experimentação.

## Fluxo Proposto

1. Carregar dados históricos dos sorteios.
2. Transformar os dados em modelos canônicos de domínio.
3. Construir features a partir do histórico recente e de longo prazo.
4. Pedir a uma ou mais estratégias que gerem jogos candidatos.
5. Ranquear os jogos usando sinais clássicos ou quânticos.
6. Executar backtest dos candidatos gerados em janelas históricas.
7. Persistir métricas e um relatório em Markdown.

## Plano Recomendado de Evolução

### Fase 1. Base determinística

- implementar regras de domínio da Lotofácil;
- carregar e validar sorteios históricos;
- criar baselines aleatórias e estatísticas;
- adicionar backtesting reproduzível.

### Fase 2. Ranqueamento orientado por features

- construir pipeline de extração de features;
- ranquear jogos candidatos com heurísticas simples;
- medir se as heurísticas superam geração aleatória ingênua.

### Fase 3. Experimentação quântica

- introduzir um adaptador para SDK quântico;
- começar por problemas experimentais pequenos;
- comparar saídas quânticas com baselines clássicas;
- documentar o que é sinal e o que é ruído.

### Fase 4. Fluxo híbrido de pesquisa

- usar filtros clássicos para reduzir o espaço de busca;
- usar rotinas quânticas para amostrar ou ranquear dentro desse espaço;
- armazenar configurações e resultados dos experimentos para reprodução futura.

## Direção Tecnológica

Para uma versão educacional, Python é a melhor escolha inicial porque oferece acesso fácil a:

- ferramental científico;
- bibliotecas de análise de dados;
- suporte a testes;
- SDKs quânticos como Qiskit.

Stack sugerida:

- Python para a implementação principal;
- `pytest` para testes;
- `pandas` para exploração de dados históricos;
- Qiskit para experimentos quânticos;
- relatórios em Markdown para saída de pesquisa.

## Riscos e Guardrails Científicos

- Previsão de loteria não é uma promessa cientificamente validada.
- Ajuste de padrões históricos pode facilmente virar overfitting.
- Computação quântica pode adicionar complexidade sem ganho mensurável.
- Resultados devem ser apresentados como experimentos, não como alegações de previsibilidade.

O projeto deve distinguir explicitamente:

- geração;
- ranqueamento;
- simulação;
- alegações de previsão.

## Recomendação Final

A melhor arquitetura para `qulotto` é uma arquitetura híbrida de pesquisa:

- um núcleo de domínio rigoroso;
- um pipeline reproduzível de dados e avaliação;
- motores de estratégia plugáveis;
- um módulo isolado de experimentação quântica.

Essa estrutura mantém o projeto honesto, extensível e útil para aprender computação quântica sem amarrar toda a base de código a suposições não comprovadas.
